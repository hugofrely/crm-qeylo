# i18n — Internationalisation FR/EN

**Date :** 2026-03-08
**Statut :** Approuvé
**Langues :** Français (défaut), English

## Décisions

| Sujet | Décision |
|-------|----------|
| Librairie frontend | `next-intl` |
| Backend | `django.utils.translation` natif |
| Routing marketing | Préfixe URL : `/fr/pricing`, `/en/pricing` |
| Routing app | Préfixe URL + cookie `NEXT_LOCALE` + préférence profil |
| Détection locale | Middleware : URL > cookie > header navigateur |
| Langue API | Header `Accept-Language` via intercepteur fetch/axios |
| Emails/PDFs | `user.preferred_language` → fallback `organization.default_language` → fallback `'fr'` |
| Clés de traduction | camelCase, hiérarchie sémantique, max 3 niveaux |
| Fichiers traduction | Un fichier JSON par namespace par langue |

### Routing

- **Middleware next-intl** : détecte la locale, redirige `/dashboard` → `/fr/dashboard`
- **Pages marketing** : `/fr/pricing`, `/en/pricing` — SEO-friendly
- **App authentifiée** : locale dans l'URL, persistée via cookie `NEXT_LOCALE`
- **Changement de langue** : mise à jour cookie + champ `preferred_language` en base + redirect

### Utilisation dans les composants

```tsx
// Server Component
import { useTranslations } from 'next-intl';

export default function ContactsPage() {
  const t = useTranslations('contacts');
  return <h1>{t('title')}</h1>;
}

// Client Component
'use client';
import { useTranslations } from 'next-intl';

export function ContactForm() {
  const t = useTranslations('contacts.form');
  return <input placeholder={t('searchPlaceholder')} />;
}
```

## Architecture Backend (Django)

### Configuration

```python
# settings.py
LANGUAGE_CODE = 'fr'
LANGUAGES = [('fr', 'Français'), ('en', 'English')]
USE_I18N = True
LOCALE_PATHS = [BASE_DIR / 'locale']

MIDDLEWARE = [
    ...
    'django.middleware.locale.LocaleMiddleware',
    ...
]
```

### Structure des fichiers

```
backend/
└── locale/
    ├── fr/LC_MESSAGES/django.po
    └── en/LC_MESSAGES/django.po
```

### Utilisation

```python
from django.utils.translation import gettext_lazy as _

# Models
class Contact(models.Model):
    class Meta:
        verbose_name = _("Contact")

# Views
raise ValidationError(_("Ce contact existe déjà."))
```

### Emails et PDFs

```python
from django.utils.translation import override as translation_override

def send_notification_email(user, ...):
    lang = user.preferred_language or user.organization.default_language or 'fr'
    with translation_override(lang):
        subject = _("Nouvelle tâche assignée")
        body = render_to_string('emails/task_assigned.html', context)
        send_mail(subject, body, ...)
```

### Modèle utilisateur

```python
# Ajout aux modèles existants
# User
preferred_language = models.CharField(
    max_length=5, choices=[('fr', 'Français'), ('en', 'English')],
    default='fr'
)

# Organization
default_language = models.CharField(
    max_length=5, choices=[('fr', 'Français'), ('en', 'English')],
    default='fr'
)
```

## Data Flow

```
1. Visiteur → middleware next-intl détecte Accept-Language → redirect /fr/ ou /en/

2. Login → frontend lit user.preferred_language → redirect si différent → maj cookie

3. Changement langue Settings → PATCH /api/accounts/me/ → maj cookie → redirect

4. Appel API → intercepteur ajoute Accept-Language → Django LocaleMiddleware active la langue

5. Email/PDF → user.preferred_language → fallback organization.default_language → fallback 'fr'
```

## Conventions de clés

```json
{
  "title": "Contacts",
  "table": {

    "email": "Email",

  "form": {

```

- camelCase pour les clés
- Hiérarchie max 3 niveaux
- Strings génériques dans `common.json`

## Stratégie de migration

Migration progressive par module :
1. Setup technique (middleware, config, routing `[locale]`)
2. `common` (boutons, labels génériques)
3. Module par module : `auth` → `settings` → `contacts` → `deals` → `tasks` → `pipeline` → `calendar` → `dashboard` → `chat` → `marketing`
4. Backend : wrapper strings avec `_()`, `makemessages`, traduire
